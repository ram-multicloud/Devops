# Ansible — Introduction, Ping Command & Default Paths

## Ansible

**Ansible** is an open-source IT automation tool used for:

| Use Case | Description |
|----------|-------------|
| **Configuration Management** | Install packages, manage files, services |
| **Application Deployment** | Deploy code across multiple servers |
| **Orchestration** | Coordinate complex multi-step workflows |
| **Provisioning** | Set up infrastructure (VMs, cloud resources) |


### Key Characteristics
- **Agentless** — No software needed on managed nodes (uses SSH)
- **Idempotent** — Running a task twice gives the same result
- **YAML-based** — Playbooks are human-readable
- **Push-based** — Control machine pushes config to nodes

---

## How Ansible Works

```
Control Node (your machine)
        │
        │  SSH + Python
        ▼
┌───────────────────────────────────┐
│         Managed Nodes             │
│  node-1   node-2   node-3  ...    │
└───────────────────────────────────┘
```

- Ansible is installed **only on the control node**
- Managed nodes need only **SSH** and **Python** installed
- No agents, no daemons on remote machines

## 3. Ansible Architecture

```
┌─────────────────────────────────────────┐
│           CONTROL NODE                  │
│                                         │
│  ┌─────────┐   ┌──────────┐             │
│  │Inventory│   │ Playbook │             │
│  │  File   │   │ (YAML)   │             │
│  └────┬────┘   └────┬─────┘             │
│       └──────┬───────┘                  │
│          ┌───▼────┐                     │
│          │Ansible │                     │
│          │ Core   │                     │
│          └───┬────┘                     │
└─────────────┼───────────────────────────┘
              │ SSH
   ┌──────────┼──────────┐
   ▼          ▼          ▼
node-1      node-2      node-3
(Python)   (Python)   (Python)
```

| Component | Role |
|-----------|------|
| **Control Node** | Machine where Ansible is installed and run |
| **Managed Nodes** | Remote machines being configured |
| **Inventory** | List of managed node IPs/hostnames |
| **Playbook** | YAML file describing automation tasks |
| **Module** | Unit of work Ansible executes (e.g., ping, apt, copy) |

## Default Paths

These are the key file and directory paths Ansible uses by default:

| Path | Purpose |
|------|---------|
| `/etc/ansible/ansible.cfg` | **Main configuration file** |
| `/etc/ansible/hosts` | **Default inventory file** |
| `/etc/ansible/roles/` | Default roles directory |
| `~/.ansible/` | User-level Ansible cache & tmp files |
| `~/.ansible/tmp/` | Temporary files during task execution |
| `/usr/bin/ansible` | Ansible binary |
| `/usr/lib/python3/dist-packages/ansible/` | Ansible library (modules) |

### Configuration File Priority (highest → lowest)

```
1. ANSIBLE_CONFIG (environment variable)
2. ./ansible.cfg  (in current directory)
3. ~/.ansible.cfg (in home directory)
4. /etc/ansible/ansible.cfg (global default)
```

> Ansible uses the **first config file found** in the above order.

### View Active Configuration

```bash
ansible --version
# Shows "config file = ..." path currently in use

ansible-config dump --only-changed
# Shows only settings changed from defaults
```

## Inventory File

The inventory file tells Ansible **which machines to manage**.

### Default Location

```
/etc/ansible/hosts
```

### Basic Inventory Format

```ini
# /etc/ansible/hosts

# Individual hosts
192.168.0.101
192.168.0.102

# Named group
[webservers]
node-1 ansible_host=192.168.0.101
node-2 ansible_host=192.168.0.102

[dbservers]
db-1 ansible_host=192.168.0.201

# Group of groups
[all_servers:children]
webservers
dbservers
```

### Inventory with SSH Key Variables

```ini
[webservers]
node-1 ansible_host=192.168.0.101 ansible_user=myuser ansible_ssh_private_key_file=~/.ssh/temp_ed25519
node-2 ansible_host=192.168.0.102 ansible_user=myuser ansible_ssh_private_key_file=~/.ssh/temp_ed25519
```

### Use a Custom Inventory File

```bash
ansible -i /path/to/my_inventory all -m ping
```

## Ansible Ping Command

The `ping` module is used to **verify Ansible can reach and communicate** with managed nodes. This is **not** an ICMP ping — it tests SSH connectivity and Python availability on the remote host.

### Syntax

```bash
ansible <host-or-group> -m ping [options]
```

### Ping All Hosts in Inventory

```bash
ansible all -m ping
```

### Ping a Specific Group

```bash
ansible webservers -m ping
```

### Ping a Single Host

```bash
ansible node-1 -m ping
```

### Ping with Custom Inventory & User

```bash
ansible all -m ping -i /etc/ansible/hosts -u myuser
```

### Ping with SSH Key

```bash
ansible all -m ping \
  -i /etc/ansible/hosts \
  -u myuser \
  --private-key ~/.ssh/temp_ed25519
```

### Expected Successful Output

```json
node-1 | SUCCESS => {
    "ansible_facts": {
        "discovered_interpreter_python": "/usr/bin/python3"
    },
    "changed": false,
    "ping": "pong"
}
```

### Failure Output (SSH issue)

```
node-1 | UNREACHABLE! => {
    "changed": false,
    "msg": "Failed to connect to the host via ssh",
    "unreachable": true
}
```


## Common Ad-Hoc Commands

Ad-hoc commands run a single module without writing a playbook.

```bash
# Syntax
ansible <hosts> -m <module> -a "<arguments>" [options]
```

| Task | Command |
|------|---------|
| Test connectivity | `ansible all -m ping` |
| Check uptime | `ansible all -m command -a "uptime"` |
| Check disk space | `ansible all -m command -a "df -h"` |
| Check memory | `ansible all -m command -a "free -m"` |
| Install a package | `ansible all -m apt -a "name=nginx state=present" --become` |
| Restart a service | `ansible all -m service -a "name=nginx state=restarted" --become` |
| Copy a file | `ansible all -m copy -a "src=/tmp/file.txt dest=/tmp/file.txt"` |
