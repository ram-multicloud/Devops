# GitHub Actions: Inputs and Types

GitHub Actions supports inputs in three different contexts, each with its own supported types and syntax:

1. `workflow_dispatch` inputs (manual trigger, UI/API-driven)
2. `workflow_call` inputs (reusable workflows)
3. Composite action inputs (`action.yml`)

---

## 1. `workflow_dispatch` Inputs

Used when a workflow is triggered manually from the Actions tab, GitHub CLI, or API.

### Supported Types

| Type          | Description                                  |
|---------------|-----------------------------------------------|
| `string`      | Free-text input                               |
| `boolean`     | Renders as a checkbox                         |
| `choice`      | Dropdown, requires `options`                  |
| `environment` | Dropdown of configured environments           |
| `number`      | Numeric input (rendered as text in UI)        |

### Syntax

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: environment
      version:
        description: 'Release version'
        required: true
        type: string
        default: 'latest'
      run_tests:
        description: 'Run test suite'
        required: false
        type: boolean
        default: true
      log_level:
        description: 'Logging level'
        required: true
        type: choice
        options:
          - debug
          - info
          - warning
          - error
        default: info
      retry_count:
        description: 'Number of retries'
        required: false
        type: number
        default: 3
```

### Accessing the Inputs

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Show inputs
        run: |
          echo "Environment: ${{ inputs.environment }}"
          echo "Version: ${{ inputs.version }}"
          echo "Run tests: ${{ inputs.run_tests }}"
          echo "Log level: ${{ inputs.log_level }}"
          echo "Retries: ${{ inputs.retry_count }}"
```

**Note:** Inside the workflow that declares `workflow_dispatch`, inputs are accessed via `inputs.<name>` (or `github.event.inputs.<name>` for raw string values from the event payload).

## 2. `workflow_call` Inputs (Reusable Workflows)

Used when one workflow is called by another via `uses:`.

### Supported Types

| Type      | Description            |
|-----------|-------------------------|
| `string`  | Text value               |
| `number`  | Numeric value             |
| `boolean` | True/false value          |

There is no `choice` or `environment` type here — validation of allowed values must be done manually inside the called workflow.

### Defining Inputs (Called Workflow)

```yaml
# .github/workflows/reusable-deploy.yml
on:
  workflow_call:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        type: string
      replicas:
        description: 'Number of replicas'
        required: false
        type: number
        default: 1
      dry_run:
        description: 'Perform a dry run'
        required: false
        type: boolean
        default: false
    secrets:
      DEPLOY_TOKEN:
        required: true

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        run: |
          echo "Environment: ${{ inputs.environment }}"
          echo "Replicas: ${{ inputs.replicas }}"
          echo "Dry run: ${{ inputs.dry_run }}"
```

### Calling the Workflow (Caller)

```yaml
# .github/workflows/caller.yml
jobs:
  call-deploy:
    uses: ./.github/workflows/reusable-deploy.yml
    with:
      environment: production
      replicas: 3
      dry_run: false
    secrets:
      DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
```

## 3. Composite Action Inputs (`action.yml`)

Used when defining a custom composite action.

### Supported Types

Composite actions support **only `string`** inputs. There is no native `boolean`, `number`, or `choice` type — all values are passed and read as strings, so type conversion/validation must be done manually in the `run:` steps.

### Syntax

```yaml
# action.yml
name: 'Deploy App'
description: 'Deploys the application to a target environment'
inputs:
  environment:
    description: 'Target environment'
    required: true
  replicas:
    description: 'Number of replicas'
    required: false
    default: '1'
  dry_run:
    description: 'Perform a dry run (true/false)'
    required: false
    default: 'false'

runs:
  using: 'composite'
  steps:
    - name: Deploy
      shell: bash
      run: |
        echo "Environment: ${{ inputs.environment }}"
        echo "Replicas: ${{ inputs.replicas }}"
        if [ "${{ inputs.dry_run }}" = "true" ]; then
          echo "Dry run mode enabled"
        fi
```

### Using the Composite Action

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: ./.github/actions/deploy-app
        with:
          environment: staging
          replicas: '2'
          dry_run: 'true'
```

## Quick Reference

| Context             | Types Available                                   | Access Syntax     |
|----------------------|-----------------------------------------------------|--------------------|
| `workflow_dispatch`  | `string`, `boolean`, `choice`, `environment`, `number` | `${{ inputs.name }}` |
| `workflow_call`      | `string`, `number`, `boolean`                        | `${{ inputs.name }}` |
| Composite action     | `string` only                                        | `${{ inputs.name }}` |

**Key gotcha:** Even where a type like `boolean` or `number` is declared (`workflow_dispatch`, `workflow_call`), values coming through `run:` shell steps are always strings. Comparisons like `if [ "${{ inputs.dry_run }}" = "true" ]` are required — `if [ ${{ inputs.dry_run }} ]` will not behave as expected.
