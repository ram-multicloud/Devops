# GitHub Actions Self-Hosted Runner — Configuration Steps

## 1. Create the runner on GitHub

Go to your repo → **Settings → Actions → Runners → New self-hosted runner**, select Linux/x64. GitHub generates a registration token and the exact commands below (the token is short-lived — copy it fresh each time).

## 2. Download and extract

```bash
mkdir actions-runner && cd actions-runner
curl -o actions-runner.tar.gz -L https://github.com/actions/runner/releases/download/v2.319.1/actions-runner-linux-x64-2.319.1.tar.gz
tar xzf ./actions-runner.tar.gz
```

## 3. Configure

```bash
./config.sh --url https://github.com/<org>/<repo> --token <REG_TOKEN>
```

Prompts:
- Runner group (default is fine unless groups are set up)
- Runner name (defaults to hostname)
- Labels (add custom ones to target this runner specifically in `runs-on:`)
- Work folder (default `_work` is fine)

## 4. Grant Docker access

```bash
sudo usermod -aG docker $USER
```

Do this **before** installing the service, so it starts with the correct group membership.

## 5. Install as a systemd service

Persistent — survives reboot/disconnect, required for long-running jobs.

```bash
sudo ./svc.sh install
sudo ./svc.sh start
```

## 6. Verify

```bash
sudo ./svc.sh status
```

In GitHub: **Settings → Actions → Runners** should show the runner as "Idle" (green).

## 7. Target the runner in a workflow

```yaml
jobs:
  build:
    runs-on: self-hosted
    timeout-minutes: 60
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v4
      - run: docker build -t myapp .
```

## Public repo considerations

- Public repos don't allow self-hosted runners in a runner group by default — must be explicitly enabled in the group's settings.
- Never target self-hosted runners from `pull_request` triggers on forked PRs — any GitHub user can trigger the workflow and run code on the machine.
- Restrict which workflows can use the runner group.
- Don't expose secrets to jobs that untrusted PRs could trigger.

## Troubleshooting: Docker permission denied

```
Error: permission denied while trying to connect to the docker API at unix:///var/run/docker.sock
```

Cause: the runner's user was added to the `docker` group *after* the service was already running — group membership only applies to new sessions.

Fix:

```bash
cd ~/actions-runner
sudo ./svc.sh stop
sudo ./svc.sh start
```

Verify:

```bash
cat /proc/$(pgrep -f Runner.Listener)/status | grep Groups
```

Confirm the `docker` group's GID (from `getent group docker`) appears in that list.
