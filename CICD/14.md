## What Is a Composite Action?

A **composite action** lets you bundle multiple workflow steps (shell commands, other actions, etc.) into a single, reusable action. Instead of copy-pasting the same sequence of steps across many workflows, you define them once in an `action.yml` file and call that action wherever needed with a single `uses:` line.

Think of it as a "function" for your workflows: it takes inputs, runs a series of steps, and can produce outputs.

**Where it lives:**
- Inside a repository, typically under `.github/actions/<action-name>/action.yml` for local/private actions.
- Or as its own standalone repository (e.g., `your-org/your-action`) if you want to publish it for reuse across repositories or publicly on the GitHub Marketplace.

---

## 2. Why Use Composite Actions?

| Problem | How Composite Actions Help |
|---|---|
| Same steps repeated in many workflows | Define once, call from anywhere |
| Long, cluttered workflow YAML files | Move logic into a dedicated, named action |
| Hard to version/test shared logic | Composite actions can be versioned (tags/releases) independently |
| Team wants a standard process (e.g., build+lint+test) | Encapsulate the standard as one action |

**Composite actions vs. Reusable Workflows** (common point of confusion):

| Aspect | Composite Action | Reusable Workflow |
|---|---|---|
| Granularity | A set of **steps** | A full **job** (or multiple jobs) |
| Called from | A step (`- uses:`) inside a job | The job level (`jobs.<id>.uses:`) |
| Can define its own `runs-on` | No — runs in the caller's job/runner | Yes — has its own `runs-on` |
| Can use secrets directly | Only via inputs passed from caller | Can declare its own `secrets:` block |
| Best for | Reusable step sequences (build, lint, notify) | Reusable full pipelines (build → test → deploy) |

---

## 3. Anatomy of `action.yml`

A composite action is defined in a file literally named `action.yml` (or `action.yaml`). Every composite action file has four main sections:

```yaml
name: 'Action Name'
description: 'What this action does'
inputs:
  # ...input definitions
outputs:
  # ...output definitions
runs:
  using: 'composite'
  steps:
    # ...the actual steps
```

### 3.1 `name`
- **Required.**
- A human-readable name shown in the GitHub UI (e.g., in the Actions run log and in the Marketplace if published).

```yaml
name: 'Build and Test App'
```

### 3.2 `description`
- **Required.**
- A short summary of what the action does. Also shown in the UI/Marketplace listing.

```yaml
description: 'Builds the application, runs unit tests, and uploads coverage reports'
```

### 3.3 `inputs`
- **Optional**, but almost always used.
- Defines the parameters the action accepts. Composite actions only support the **`string`** type — there is no native `boolean`, `number`, or `choice`. All values arrive (and must be handled) as strings.

Each input can have:

| Field | Required | Purpose |
|---|---|---|
| `description` | Recommended | Explains what the input is for |
| `required` | Optional (default `false`) | Whether the caller must supply it |
| `default` | Optional | Value used if the caller doesn't provide one |
| `deprecationMessage` | Optional | Warns callers the input is being phased out |

```yaml
inputs:
  environment:
    description: 'Target deployment environment'
    required: true
  node-version:
    description: 'Node.js version to use'
    required: false
    default: '20'
  enable-cache:
    description: 'Whether to cache dependencies (true/false, passed as string)'
    required: false
    default: 'true'
```

### 3.4 `outputs`
- **Optional.**
- Defines values the action produces that the calling workflow can use in later steps.
- Each output's `value` is normally pulled from a specific step's output using the step's `id`.

```yaml
outputs:
  build-status:
    description: 'Result of the build step'
    value: ${{ steps.build.outputs.status }}
  artifact-path:
    description: 'Path to the built artifact'
    value: ${{ steps.build.outputs.path }}
```

For this to work, the referenced step must set an output, e.g.:

```yaml
steps:
  - name: Build
    id: build
    shell: bash
    run: |
      echo "status=success" >> "$GITHUB_OUTPUT"
      echo "path=dist/app.tar.gz" >> "$GITHUB_OUTPUT"
```

### 3.5 `runs`
- **Required.**
- Tells GitHub how to execute the action. For composite actions, this is always:

```yaml
runs:
  using: 'composite'
  steps:
    - ...
```

(As opposed to `using: 'node20'` for JavaScript actions, or `using: 'docker'` for Docker container actions — composite actions are a distinct action type.)

## 4. The `steps` Block (Core of the Composite Action)

This is where the actual work happens. It looks almost identical to steps in a normal workflow job, with **one key difference**: every step that runs shell commands **must** explicitly declare `shell:`.

```yaml
runs:
  using: 'composite'
  steps:
    - name: Install dependencies
      shell: bash
      run: npm install

    - name: Run tests
      shell: bash
      run: npm test

    - name: Use another action
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
```

### 4.1 Why `shell:` is mandatory
In a normal workflow job, `shell` is optional because it's inherited from the runner's OS default (`bash` on Linux/macOS, `pwsh` on Windows). Inside a **composite action**, GitHub does not assume a default — you must specify it on every `run:` step, or the action will fail to validate.

Common values: `bash`, `pwsh`, `python`, `sh`, `cmd`.

### 4.2 Referencing inputs inside steps
Use the `inputs.<name>` context:

```yaml
- name: Deploy
  shell: bash
  run: |
    echo "Deploying to ${{ inputs.environment }}"
```

### 4.3 Using other actions inside a composite action
A composite action's steps can call other actions (including other composite actions), just like a normal job:

```yaml
- name: Checkout
  uses: actions/checkout@v4

- name: Setup Node
  uses: actions/setup-node@v4
  with:
    node-version: ${{ inputs.node-version }}
```

### 4.4 Step IDs and outputs
Give a step an `id:` if you need to reference its output later (either in a later step, or in the action's top-level `outputs:` block):

```yaml
- name: Get version
  id: version
  shell: bash
  run: echo "value=$(cat VERSION)" >> "$GITHUB_OUTPUT"

- name: Print version
  shell: bash
  run: echo "Version is ${{ steps.version.outputs.value }}"
```

### 4.5 Conditionals (`if:`)
Steps inside a composite action support `if:` just like normal workflow steps:

```yaml
- name: Notify on failure
  if: failure()
  shell: bash
  run: echo "Build failed!"
```

### 4.6 Environment variables
You can set `env:` at the step level (or reference `inputs.*` inside `env:`):

```yaml
- name: Build
  shell: bash
  env:
    NODE_ENV: production
    TARGET: ${{ inputs.environment }}
  run: npm run build
```

## 5. Full Example: `action.yml`

```yaml
name: 'Build, Test, and Report'
description: 'Installs dependencies, builds the app, runs tests, and reports status'

inputs:
  node-version:
    description: 'Node.js version to use'
    required: false
    default: '20'
  run-lint:
    description: 'Whether to run the linter (true/false)'
    required: false
    default: 'true'

outputs:
  test-result:
    description: 'Outcome of the test step'
    value: ${{ steps.test.outputs.result }}

runs:
  using: 'composite'
  steps:
    - name: Setup Node
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}

    - name: Install dependencies
      shell: bash
      run: npm ci

    - name: Lint
      if: inputs.run-lint == 'true'
      shell: bash
      run: npm run lint

    - name: Run tests
      id: test
      shell: bash
      run: |
        npm test
        echo "result=passed" >> "$GITHUB_OUTPUT"
```

## 6. Calling the Composite Action

### 6.1 Local action (same repository)

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build and Test
        id: build-test
        uses: ./.github/actions/build-test
        with:
          node-version: '20'
          run-lint: 'true'

      - name: Show result
        run: echo "Test result: ${{ steps.build-test.outputs.test-result }}"
```

- The path after `uses:` (`./.github/actions/build-test`) points to the **directory** containing `action.yml`, not the file itself.
- A `checkout` step is required first — local composite actions are read from the checked-out repository content.

### 6.2 External action (separate repository)

```yaml
steps:
  - uses: your-org/your-action@v1
    with:
      node-version: '20'
```

- References `owner/repo@ref`, where `ref` is a tag, branch, or commit SHA.
- Pinning to a tag (e.g., `@v1`) or SHA is recommended for stability and supply-chain security over using a branch like `@main`.

## 7. Key Rules and Gotchas

1. **Only `string` inputs are supported.** Booleans and numbers must be passed as strings (`'true'`, `'3'`) and explicitly parsed/compared in `run:` steps (e.g., `if [ "${{ inputs.flag }}" = "true" ]`).
2. **`shell:` is required on every `run:` step.** Omitting it causes a validation error — unlike normal jobs, there's no OS-default fallback.
3. **No `runs-on`.** A composite action does not choose its own runner; it always executes on the runner of the job that calls it.
4. **No top-level `env:` or `permissions:` in the action itself** that apply globally — environment variables are set per-step, and permissions are controlled by the caller's workflow.
5. **Secrets are not inherited automatically.** A composite action cannot access the caller's secrets unless they are explicitly passed in as `inputs`.
6. **File path matters for local actions.** `uses: ./.github/actions/<name>` must point to the directory containing `action.yml`, and the repository must be checked out first.
7. **Outputs must be wired through `steps.<id>.outputs.<name>`.** Simply setting a variable inside a step isn't enough — it must be written to `$GITHUB_OUTPUT` and referenced correctly in the action's `outputs:` block.
8. **Nesting is allowed.** A composite action can call another composite action as one of its steps.
9. **Versioning matters for external actions.** Use tags or commit SHAs (not floating branches) when referencing composite actions from other repositories, to avoid unexpected breaking changes.

---

## 8. Quick Reference Summary

| Section | Required | Purpose |
|---|---|---|
| `name` | Yes | Display name |
| `description` | Yes | What the action does |
| `inputs` | No (usually yes) | Parameters (string only) |
| `outputs` | No | Values exposed to the caller |
| `runs.using` | Yes | Must be `'composite'` |
| `runs.steps` | Yes | The actual step sequence; `shell:` required on every `run:` step |
